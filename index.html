Security Architecture

Enterprise-Grade Security
for South African Treasury

OneBank TMS is built with security at every layer — from bcrypt-hashed passwords and JWT sessions to API-level role enforcement and a complete audit trail.

Security Controls

Defence in Depth

Multiple independent security layers ensure your treasury data is protected at rest, in transit, and at the application level.

🔑
Authentication

NextAuth.js v5 with credentials provider. JWT tokens with server-side validation. Session-based authentication with secure cookie handling.

NextAuth.js v5

🔒
Password Security

All passwords are hashed using bcrypt with cost factor 12 — the industry standard. Plain-text passwords are never stored or logged anywhere in the system.

bcrypt cost factor 12

🛡️
Route Protection

proxy.ts middleware enforces authentication on all dashboard routes. requireAuth() and requireRole() guards protect every sensitive API endpoint.

Centralised enforcement

🌐
Transport Security

TLS/HTTPS enforced by Vercel. HTTP requests automatically redirect to HTTPS. Neon PostgreSQL database connection requires SSL (sslmode=require).

TLS enforced

📋
Security Headers

HSTS (max-age=63072000), X-Frame-Options: SAMEORIGIN, X-Content-Type-Options: nosniff, Referrer-Policy, Permissions-Policy, and DNS-Prefetch-Control — all applied via next.config.ts.

OWASP hardened

📝
Complete Audit Trail

ActivityLog model tracks every system event — action type, description, user name, entity, IP address, status, and full timestamp. Searchable and filterable in the admin UI.

Full audit log

⏱️
Rate Limiting

Upstash Redis-backed rate limiting on authentication endpoints — 10 requests per 10 seconds per IP. Prevents brute force attacks in production with graceful local fallback.

Upstash Redis

💰
Transfer Limits

Role-based payment amount restrictions enforced at API level — daily, per-transaction, and monthly limits configurable per user role to prevent unauthorised high-value payments.

Role-enforced

📱
QR Pay Authentication

Configurable authentication method for QR payments — choose from Biometric, PIN, Password, or OTP. Security level indicator shows risk rating per method selected.

Configurable

Security & Compliance

Compliance & Data

OneBank TMS is a software platform, not a financial services provider — we do not hold or move funds. We are therefore not currently required to be licensed by the SARB. A SARB payment service provider licence is on our roadmap for Q4 2026 as we move toward live payment execution.

Yes. We process personal and financial data under POPIA (Protection of Personal Information Act, 2013). We have a formal POPIA Notice, Privacy Policy, and Terms of Service. Our data is hosted on encrypted EU-region servers (Neon PostgreSQL) with adequate safeguards for cross-border transfers under POPIA section 72. See our full POPIA notice in the footer.

Application data is stored on Neon PostgreSQL, hosted in the EU region with GDPR-compliant infrastructure. The application runs on Vercel's edge network. All data in transit is protected by TLS 1.2+, and sensitive fields are encrypted at rest with AES-256.

We are currently in Beta and have implemented OWASP-hardened security controls (bcrypt cost factor 12, JWT sessions, rate limiting, RBAC enforced at API level, full audit trail). ISO 27001 certification and SOC 2 Type II audit are planned for Q4 2026. We are transparent about where we are on the certification journey.

We operate a responsible disclosure policy. If you discover a potential security issue, please contact security@onebank.co.za with details. We commit to acknowledging your report within 48 hours. We do not pursue legal action against good-faith security researchers.

OneBank TMS does not execute payments during Beta (simulation mode only), so FICA and POCA obligations that apply to payment service providers do not currently apply. Our Terms of Service explicitly prohibit use of the platform for money laundering or any FICA/POCA-prohibited activities.
